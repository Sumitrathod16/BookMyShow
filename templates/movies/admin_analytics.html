Performance Optimization & Consistency Architecture

How Race Conditions are Prevented

During concurrent seat selections, database integrity is protected by performing explicit pessimistic row-level locks using Django's select_for_update() database expression within an atomic transaction.

When a request arrives, the row is locked exclusively, forcing simultaneous requests for the same seat to wait. Once the lock is released, subsequent requests double-check the seat status (which has now transitioned to `is_booked = True`) and abort with an IntegrityError.

Database Aggregation & Cache Consistency

Database Aggregation: Queries calculate totals (like revenue and occupancy rate) entirely at the database layer using SQL SUM, COUNT, and AVG functions. This avoids loading large datasets into Django application memory.

Caching Strategy: The analytics API uses Django's local memory caching backend (LocMemCache) configured for 60 seconds. This avoids query performance degradation during high-traffic intervals while ensuring near-real-time updates.
